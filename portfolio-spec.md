# Portfolio Product Spec — Mithun Arun

## 0. Purpose

This document defines the product direction, information architecture, content strategy, technical feature set, and quality bar for Mithun Arun's personal website.

The website should present Mithun as a high-potential AI/CS researcher, software engineer, product builder, startup-oriented founder, and student leader. It must be optimized for recruiters and technical reviewers in three target areas:

1. **Software Engineering** — evidence of strong engineering fundamentals, full-stack/mobile development, maintainable systems, product execution, and ability to ship.
2. **AI / Machine Learning** — evidence of research ability, model development, signal processing, healthcare AI experience, publications, and technical communication.
3. **Quant Tech / Data-Driven Engineering** — evidence of mathematical thinking, statistical validation, time-series/signal analysis, performance metrics, experimentation, and rigorous problem-solving.

The portfolio should not feel like a decorative resume. It should feel like a polished technical product that proves Mithun can research, build, explain, lead, and iterate.

Requirements:

- Answers must be grounded only in site content.
- Answers should link to relevant portfolio pages.

### 7.2 Required Animation Standards

- Use tasteful motion, not distracting effects.
- Use Framer Motion for page transitions, section reveals, card hover effects, timeline animations, modal transitions, and small interaction states.
- Support `prefers-reduced-motion`.
- Animations must not reduce accessibility, readability, or performance.
- Interactive elements must work on desktop and mobile.
- Motion should reinforce meaning: reveal hierarchy, show state changes, and guide attention.
- Avoid animation just for decoration when it slows down the site.

At least two of the following technical visuals must be implemented in Version 1. More can be added in later versions.

#### Neural Network Background

A subtle animated node-and-edge visual in the hero section.

Purpose:

- Signals AI/ML identity immediately.
- Creates a technical first impression.

#### EEG/ECG Signal Renderer

An animated waveform representing Mithun's AI healthcare research.

Purpose:

- Connects visual design directly to Mithun's research.
- Helps the site feel specific rather than generic.

#### Research Pipeline Diagram

An interactive diagram showing:

1. Data input
2. Signal preprocessing
3. Feature extraction
4. Model prediction
5. Clinical interpretation

Purpose:

- Makes the research easier to understand.
- Shows systems thinking and ML workflow knowledge.

#### Quant/Math Visual Layer

Subtle visuals involving graphs, probability, optimization, thresholds, or time-series signals.

Purpose:

- Appeals to quant-tech recruiters.
- Reinforces data-driven and mathematical thinking.

#### 3D or Canvas-Based Technical Visual

A React Three Fiber, Canvas, or SVG-based technical visual may be used if performance remains strong.

Purpose:

- Adds technical impressiveness.
- Demonstrates frontend engineering ability.

### 7.4 Performance Requirements for Animation and Rendering

- Animations must not hurt Lighthouse performance.
- Avoid heavy 3D assets unless optimized.
- Lazy-load expensive visual components.
- Keep first contentful paint fast.
- Prefer SVG, Canvas, CSS, and lightweight animation before complex 3D.
- Ensure visual effects do not block access to core content.
- The site must still feel fast on a typical laptop and phone.

### 7.5 Recruiter Impact of Interactivity

The interactive layer should help recruiters reach the right evidence faster:

- SWE recruiters should quickly see shipped software, architecture, and product execution.
- AI/ML recruiters should quickly see research methods, metrics, and publications.
- Quant-tech recruiters should quickly see signal processing, statistical validation, threshold analysis, and data-heavy work.

Interactivity should never hide important proof. Core claims must remain visible through standard page content even if a user does not interact deeply.

## 8. Recommended Tech Stack

### 8.1 Core

- **Framework:** Next.js App Router
- **Language:** TypeScript
- **Styling:** Tailwind CSS
- **Component System:** shadcn/ui
- **Animation:** Framer Motion
- **Content:** MDX + local JSON/YAML content files
- **Deployment:** Vercel

### 8.2 AI and Search

- **AI Chat:** Vercel AI SDK or equivalent LLM integration
- **Knowledge Base:** Local MDX/JSON content converted into structured chunks
- **Search:** Fuse.js for local search initially; upgrade to vector search only if needed

### 8.3 Data Visualization

- **Charts:** Recharts or Tremor
- **Custom Technical Visuals:** React + SVG + Canvas where needed
- **Signal Visualizations:** Precomputed sample EEG/ECG traces or synthetic demo traces if real data cannot be shown

### 8.4 Visual and Animation Stack

- **Framer Motion:** Primary animation library for transitions, reveals, hover states, modals, timeline interactions, and microinteractions.
- **SVG / Canvas:** Preferred for EEG, ECG, neural-network, data-flow, and quant/math visualizations.
- **React Three Fiber:** Optional for 3D effects only if performance remains strong and the effect adds clear value.
- **Recharts or Tremor:** Recommended for research metrics, model performance charts, and recruiter-friendly data visualizations.
- **CSS Transitions:** Use for lightweight hover states and simple interaction polish.

### 8.5 Content and Credibility

- **Writing:** MDX posts
- **Resume:** Static PDF in `/public`
- **Project data:** `/content/projects.ts` or `/content/projects.json`
- **Research data:** `/content/research.ts` or `/content/research.json`
- **Timeline data:** `/content/timeline.ts`

### 8.6 Analytics and Monitoring

- **Analytics:** Vercel Analytics or Plausible
- **Error Monitoring:** Sentry optional
- **SEO:** Next.js metadata API, sitemap, robots.txt, Open Graph images
